WhatsApp Flows JSON UI Limits & Validation

When designing WhatsApp Flows, you must adhere to technical constraints regarding screen counts, UI components, and JSON file sizes.

Max Number of Screens

A single Flow is strictly limited to a maximum of 100 screens.

  • Best Practice: Limit a single Flow to 3–5 screens to prevent user fatigue.
  • Modularization: If your workflow is complex, split it into smaller flows and link them using the flow_ref action.

Supported UI Components and Limits

  • Dropdown Components: Supported up to a maximum of 200 options per dropdown.
  • Embedded Links: Supported up to a maximum of 2 embedded links per screen.
  • Data Endpoint Latency: Any dynamic data fetching from an endpoint must complete within the 10-second timeout window.

Character Limits and File Sizes

  • Flow JSON Size: The total size of your Flow JSON file must not exceed 10 MB.
  • Input Constraints: Use min-chars and max-chars attributes in TextInput and TextArea to validate user input natively.
  • Template Constraints: The template message body that triggers the flow is limited to 1,024 characters, and the subtitle is limited to 80 characters.

UI Validations

✅ Valid: Using min-chars="5" and max-chars="50" on a text input to ensure the user provides a valid short description before submitting.

❌ Invalid: Attempting to render a dropdown component with 500 options, which exceeds the 200-option limit and causes the Flow validation to fail.

✅ Valid: Splitting a 20-screen onboarding process into 4 separate Flows connected via flow_ref.

❌ Invalid: Building a single JSON Flow file with 150 screens, resulting in an upload rejection.\n\n

حدود واجهة مستخدم JSON والتحقق في تدفقات واتساب

عند تصميم تدفقات واتساب (WhatsApp Flows)، يجب الالتزام بالقيود الفنية المتعلقة بعدد الشاشات، ومكونات واجهة المستخدم، وأحجام ملفات JSON.

أقصى عدد للشاشات

يقتصر التدفق الواحد بشكل صارم على حد أقصى يبلغ 100 شاشة.

  • أفضل الممارسات: يوصى بقصر التدفق الواحد على 3-5 شاشات لمنع إرهاق المستخدم.
  • التجزئة: إذا كان سير العمل معقداً، قم بتقسيمه إلى تدفقات أصغر واربطها باستخدام إجراء flow_ref.

مكونات واجهة المستخدم المدعومة والحدود

  • القوائم المنسدلة (Dropdowns): مدعومة بحد أقصى 200 خيار لكل قائمة منسدلة.
  • الروابط المضمنة (Embedded Links): مدعومة بحد أقصى رابطين مضمنين في كل شاشة.
  • زمن انتقال نقطة بيانات النهاية: يجب أن يكتمل أي جلب بيانات ديناميكي من نقطة النهاية خلال نافذة المهلة الزمنية البالغة 10 ثوانٍ.

حدود الأحرف وأحجام الملفات

  • حجم ملف Flow JSON: يجب ألا يتجاوز الحجم الإجمالي لملف JSON الخاص بالتدفق 10 ميجابايت.
  • قيود الإدخال: استخدم سمات min-chars و max-chars في TextInput و TextArea للتحقق من صحة إدخال المستخدم محلياً.
  • قيود القالب (Template): يقتصر نص رسالة القالب التي تبدأ التدفق على 1,024 حرفاً، بينما يقتصر العنوان الفرعي على 80 حرفاً.

تحققات واجهة المستخدم

✅ Valid: استخدام min-chars="5" و max-chars="50" في إدخال نص للتأكد من أن المستخدم يقدم وصفاً قصيراً صالحاً قبل الإرسال.

❌ Invalid: محاولة عرض مكون قائمة منسدلة يحتوي على 500 خيار، مما يتجاوز حد الـ 200 خيار ويؤدي إلى فشل التحقق من صحة التدفق.

✅ Valid: تقسيم عملية تأهيل (Onboarding) مكونة من 20 شاشة إلى 4 تدفقات منفصلة متصلة عبر flow_ref.

❌ Invalid: بناء ملف JSON لتدفق واحد يحتوي على 150 شاشة، مما يؤدي إلى رفض الرفع (Upload Rejection).\n\n

⚖️ Comparison

TextInput vs Dropdown

  • TextInput: Allows free-form text entry. Best for names, addresses, or specific IDs.
  • Dropdown: Restricts user to a predefined list of options. Prevents typos and parsing errors.
